# 量化交易系统架构建议

## 一、分层架构总览

```
┌─────────────────────────────────────────────────────────┐
│                        报告层                          │
│            (报表 / 绩效分析 / 归因分析)                 │
└──────────────────────┬──────────────────────────────────┘
                       │
┌──────────────────────▼──────────────────────────────────┐
│                        风控层                          │
│      (实时风控检查 / 止盈止损 / 仓位限额 / 熔断)        │
└──────────────────────┬──────────────────────────────────┘
                       │
┌──────────────────────▼──────────────────────────────────┐
│                        执行层                          │
│    (订单路由 / 算法执行 / 柜台对接 / 成交管理)          │
└──────────────────────┬──────────────────────────────────┘
                       │
┌──────────────────────▼──────────────────────────────────┐
│                        回测层                          │
│      (历史回测 / 参数优化 / 样本外测试 / 模拟撮合)      │
└──────────────────────┬──────────────────────────────────┘
                       │
┌──────────────────────▼──────────────────────────────────┐
│                        策略层                          │
│        (因子计算 / 信号生成 / 组合管理 / 仓位分配)      │
└──────────────────────┬──────────────────────────────────┘
                       │
┌──────────────────────▼──────────────────────────────────┐
│                        数据层                          │
│    (行情接入 / 数据清洗 / 存储 / 分发 / 参考数据)       │
└─────────────────────────────────────────────────────────┘
```

## 二、各层职责与技术选型

| 层 | 职责 | 关键技术选型 |
|---|---|---|
| **数据层** | 行情(Tick/K线)接入、清洗、存储、分发；参考数据(财务/因子)管理 | **存储：** InfluxDB/KDB+(时序)、PostgreSQL(结构化)、Parquet/HDF5(批量) |
| **策略层** | 因子计算、信号生成、组合优化、仓位分配 | **框架：** Python (Pandas/NumPy/Scikit-learn)；**高频可选** C++/Rust |
| **回测层** | 历史回测、参数优化、样本内外测试、模拟撮合 | **框架：** Backtrader / vn.py / Zipline；**ML：** PyTorch/TensorFlow |
| **执行层** | 订单路由、算法执行(拆单/TWAP/VWAP)、柜台对接(CTP/XTP)、成交管理 | **语言：** C++(HFT)、Go(中高频)、Python(中低频)；**协议：** FIX / 券商API |
| **风控层** | 实时风控检查(净值/敞口/杠杆)、止盈止损、仓位限额、熔断机制 | **引擎：** C++(实时)+Python(规则)；**规则引擎：** Drools / 自研DSL |
| **报告层** | 绩效报表、归因分析、夏普/最大回撤等指标、可视化 | **可视化：** Grafana + Prometheus；**报表：** Streamlit / Metabase |

## 三、模块间通信方式

1. **事件驱动 + 消息队列（推荐）**
   - **Kafka** — 海量行情分发、跨层异步解耦，适合日志/订单状态/信号流
   - **Redis Pub/Sub** — 低延迟实时行情广播，适合同机轻量通信
   - **共享内存** — 纳秒级通信，用于同机 HFT 场景（C++/Go 实现）

2. **API / RPC**
   - gRPC — 策略层↔回测层、策略层↔执行层的同步调用
   - REST — 管理层面（报告查询、配置下发、风控规则热加载）

3. **数据库轮询** — 不推荐用于实时环节，仅用于报表/离线分析

## 四、数据流示意图（文字）

```
交易所 ──▶ 数据层(清洗/存储/分发)
              │
              ▼ (Kafka Market Data Topic)
     ┌────────┴────────┐
     │  策略层(因子/信号) │
     └────────┬────────┘
              │ (Signal Topic / gRPC)
     ┌────────┴────────┐
     │  风控层(实时检查) │◄──── 风控规则配置
     └────────┬────────┘
              │ (风控通过信号)
     ┌────────┴────────┐
     │  执行层(订单路由) │──▶ 交易所(CTP/FIX)
     └────────┬────────┘
              │ (成交回报)
     ┌────────┴────────┐
     │  报告层(绩效/归因)│──▶ 数据库/可视化
     └─────────────────┘

回测流程: 历史数据 ──▶ 回测层(模拟演算) ──▶ 策略层(优化)
```

## 五、核心理念

- **单向依赖：** 上层依赖下层接口，下层不依赖上层
- **异步解耦：** 通过消息队列实现层间松耦合，支持水平扩展
- **多语言混合：** Python 快速迭代策略；C++/Go 保障执行层性能
- **风控前置：** 风控层在策略信号进入执行层之前进行拦截，确保安全
